💡 ทำไมต้องรู้จัก MQTT?

HTTP ส่งข้อมูลได้ดี แต่สำหรับอุปกรณ์ IoT ที่ต้องการสื่อสารแบบ Real-time หลายตัวพร้อมกัน พลังงานต่ำ หรือเครือข่ายไม่เสถียร — MQTT คือตัวเลือกที่ดีกว่า

MQTT คืออะไร?

MQTT ย่อมาจาก Message Queuing Telemetry Transport

เป็นโปรโตคอลสื่อสารน้ำหนักเบา ออกแบบมาตั้งแต่ปี 1999 สำหรับอุปกรณ์ที่มี:

  • แบนด์วิดธ์จำกัด (Low Bandwidth)
  • พลังงานน้อย (Low Power)
  • การเชื่อมต่อที่ไม่เสถียร (Unreliable Network)

ปัจจุบันกลายเป็นมาตรฐานหลักของ IoT ใช้ในระบบ Smart Home, โรงงาน, ยานพาหนะ และอีกมากมาย

🏗️ สถาปัตยกรรม Publish-Subscribe

┌─────────────┐    publish     ┌──────────────┐    deliver    ┌─────────────┐
│             │ ─────────────> │              │ ────────────> │             │
│  Publisher  │                │    BROKER    │               │ Subscriber  │
│  (ESP32 /   │                │  (RPi หรือ  │               │  (App /     │
│  Nano 33)   │                │   Cloud)     │               │   PC / อื่น)│
│             │                │              │               │             │
└─────────────┘                └──────────────┘               └─────────────┘
   ส่งข้อมูล                    กลางกระจายข้อมูล                  รับข้อมูล

Publish-Subscribe ต่างจาก Request-Response อย่างไร?

🌐 HTTP (Request-Response)

  • Client ต้องรู้ว่า Server อยู่ที่ไหน
  • ส่งคำขอ → รอรับคำตอบ
  • 1 ผู้ส่ง → 1 ผู้รับ
  • ต้องรอ Response ทุกครั้ง

📡 MQTT (Publish-Subscribe)

  • Publisher ไม่รู้จัก Subscriber
  • ส่งข้อมูล → ไม่ต้องรอ
  • 1 ผู้ส่ง → หลายผู้รับพร้อมกัน
  • Broker จัดการกระจายข้อมูล
📌 ข้อดีของ Decoupling

Publisher และ Subscriber ไม่จำเป็นต้องออนไลน์พร้อมกัน Broker จะเก็บข้อความไว้ให้ Subscriber ที่ออฟไลน์อยู่ และส่งให้เมื่อกลับมาออนไลน์

🔩 ส่วนประกอบหลัก

Broker — ตัวกลาง

Broker คือหัวใจของระบบ MQTT ทำหน้าที่:

  • รับข้อความจาก Publisher ทุกตัว
  • กรองและส่งต่อให้ Subscriber ที่ Subscribe Topic นั้น
  • เก็บข้อความไว้ถ้า Subscriber ออฟไลน์ (ขึ้นอยู่กับ QoS)
  • จัดการ Authentication และ Authorization
✅ Broker ยอดนิยม:
  • Mosquitto — โอเพนซอร์ส ติดตั้งบน Raspberry Pi ได้
  • HiveMQ — Cloud-based มี Free tier
  • AWS IoT Core — Managed service บน AWS
  • EMQX — High-performance สำหรับ Enterprise

Publisher — ผู้ส่ง

อุปกรณ์ที่ ส่ง (publish) ข้อมูลไปยัง Broker บน Topic ที่กำหนด

  • ไม่รู้ว่ามี Subscriber กี่คน
  • กำหนด Topic, QoS และ Retain flag ได้
  • ตัวอย่าง: Arduino Nano 33 IoT ส่งค่าอุณหภูมิทุก 5 วินาที
Python (paho-mqtt)
import paho.mqtt.client as mqtt

client = mqtt.Client()
client.connect("broker.local", 1883)

# ส่งค่า temperature ไปที่ topic
client.publish("home/sensor/temperature", "28.5")
client.disconnect()

Subscriber — ผู้รับ

อุปกรณ์ที่ สมัครรับ (subscribe) Topic ที่สนใจ Broker จะส่งข้อมูลให้ทุกครั้งที่มี Publisher ส่งมา

  • Subscribe หลาย Topic พร้อมกันได้
  • ใช้ Wildcards เพื่อรับ Topic ที่ match pattern
  • ตัวอย่าง: แอปบนโทรศัพท์แสดงค่า sensor แบบ Real-time
Python (paho-mqtt)
import paho.mqtt.client as mqtt

def on_message(client, userdata, msg):
    print(f"Topic: {msg.topic}")
    print(f"Data:  {msg.payload.decode()}")

client = mqtt.Client()
client.on_message = on_message
client.connect("broker.local", 1883)

# รับทุก topic ใต้ home/sensor/
client.subscribe("home/sensor/#")
client.loop_forever()

🗂️ Topics และ Wildcards

Topic คืออะไร?

Topic เป็นเส้นทาง (path) ลำดับชั้น คั่นด้วย / ใช้บอกว่าข้อมูลนี้คืออะไรและมาจากไหน

home/kitchen/temperature
home/bedroom/humidity
factory/line1/sensor/pressure
vehicle/truck01/speed

Wildcards

Wildcard ความหมาย ตัวอย่าง รับ Topic ใดได้บ้าง
+ แทนได้ 1 ระดับ home/+/temperature home/kitchen/temperature, home/bedroom/temperature
# แทนได้หลายระดับ (ใส่ท้ายเท่านั้น) home/# home/kitchen/temperature, home/bedroom/humidity, home/any/deep/path
ตัวอย่าง Wildcard
# Subscribe: home/+/temperature
# รับได้:    home/kitchen/temperature  ✅
# รับได้:    home/bedroom/temperature  ✅
# ไม่รับ:    home/kitchen/sensor/temp  ❌ (เกิน 1 ระดับ)
# ไม่รับ:    factory/line1/temperature ❌ (ต่างกลุ่ม)

# Subscribe: home/#
# รับได้:    home/kitchen/temperature  ✅
# รับได้:    home/bedroom/humidity     ✅
# รับได้:    home/any/deep/path/here   ✅
# ไม่รับ:    factory/line1/sensor      ❌
⚠️ ข้อควรระวัง:

Topic ขึ้นต้นด้วย $ (เช่น $SYS/) สงวนไว้สำหรับ Broker ไม่ควร publish ไปที่นั่น

📶 Quality of Service (QoS)

QoS กำหนดระดับการรับประกันว่าข้อความจะถึงผู้รับหรือไม่ มี 3 ระดับ:

QoS ชื่อ การรับประกัน เหมาะกับ
0 At most once ส่งครั้งเดียว ไม่ยืนยัน (อาจหาย) ค่า sensor ที่ส่งถี่ ไม่ critical
1 At least once รับประกันว่าถึง แต่อาจซ้ำ คำสั่งควบคุมทั่วไป
2 Exactly once รับประกันถึงครั้งเดียวเท่านั้น คำสั่งที่ต้องแม่นยำ เช่น เปิด/ปิดระบบ 
QoS 0:  Sender ─── msg ──────────────────────────────> Receiver
        (ไม่รู้ว่าถึงไหม)

QoS 1:  Sender ─── msg ──> Receiver
               <── ACK ─── Receiver
        (ถ้าไม่ได้ ACK จะส่งซ้ำ → อาจซ้ำ)

QoS 2:  Sender ─── msg ──> Receiver
               <── ACK ─── Receiver
        Sender ─── REL ──> Receiver
               <── COMP ── Receiver
        (4-way handshake → ถึงแน่นอนครั้งเดียว)

✨ คุณสมบัติสำคัญ

Retained Messages

Broker เก็บข้อความล่าสุดของแต่ละ Topic ไว้ เมื่อมี Subscriber ใหม่ Subscribe จะได้รับข้อความล่าสุดทันที โดยไม่ต้องรอ Publisher ส่งใหม่

Python
# ส่งพร้อม retain=True
client.publish("home/sensor/temperature", "28.5", retain=True)

# Subscriber ใหม่ที่ subscribe topic นี้จะได้ "28.5" ทันที

Last Will and Testament (LWT)

เมื่ออุปกรณ์หลุดออฟไลน์กะทันหัน (ไม่ได้ disconnect ปกติ) Broker จะส่งข้อความ "พินัยกรรม" ที่กำหนดไว้แทน ช่วยให้ระบบรู้ว่าอุปกรณ์ตายแล้ว

Python
# ตั้งค่า LWT ก่อน connect
client.will_set(
    topic="home/nano33/status",
    payload="offline",
    qos=1,
    retain=True
)
client.connect("broker.local", 1883)

# เมื่อ connect สำเร็จ ส่งสถานะ online
client.publish("home/nano33/status", "online", retain=True)

Security

Port การเข้ารหัส ใช้เมื่อ
1883 ไม่เข้ารหัส Network ภายใน (LAN) ทดสอบ
8883 TLS/SSL Production, อินเทอร์เน็ต

นอกจากนี้ยังรองรับ Username/Password Authentication และ Client Certificate

⚖️ เปรียบเทียบ MQTT vs HTTP

คุณสมบัติ HTTP MQTT
รูปแบบการสื่อสาร Request-Response Publish-Subscribe
น้ำหนัก (Overhead) หนักกว่า (Headers ใหญ่) เบามาก (2 bytes header)
การเชื่อมต่อ Stateless (ทุก request ใหม่) Persistent Connection
Real-time ต้อง polling หรือ WebSocket รองรับโดยตรง
หลาย Receiver ส่งทีละคน (1-to-1) ส่งพร้อมกันทุกคน (1-to-many)
เหมาะกับ Web API, REST, Webhook IoT Sensor, Real-time Event 
HTTP:  Nano 33 ──POST──> FastAPI Server
               <── 200 ── FastAPI Server    (ต้องรอ Response)

MQTT:  Nano 33 ──publish──> Broker ──> Subscriber A
                                    ──> Subscriber B
                                    ──> Subscriber C
       (ส่งครั้งเดียว ถึงทุกคนที่ Subscribe)
✅ เลือกใช้อะไร?
  • ใช้ HTTP เมื่อ: ต้องการ REST API, web integration, ต้องการ response ทันที
  • ใช้ MQTT เมื่อ: Real-time, หลายอุปกรณ์, battery-powered, เครือข่ายไม่เสถียร

🌍 Use Cases ใน IoT

สถานการณ์ ใช้ MQTT เพราะ
Smart Home อุปกรณ์หลายตัวสื่อสารกันแบบ Real-time, แอปควบคุมได้ทันที
โรงงาน (IIoT) รับค่า sensor หลายจุดพร้อมกัน แจ้งเตือนทันทีเมื่อผิดปกติ
ยานพาหนะ เครือข่ายไม่เสถียร MQTT จัดการ reconnect และ QoS ได้ดี
Smart Agriculture ประหยัดแบตเตอรี่ แบนด์วิดธ์น้อย ส่งข้อมูล sensor เป็นระยะ

✅ เช็คความเข้าใจ

1. MQTT ต่างจาก HTTP อย่างไรในด้านรูปแบบการสื่อสาร?
ดูเฉลย

HTTP ใช้รูปแบบ Request-Response (Client ต้องขอก่อนถึงจะได้รับ) ส่วน MQTT ใช้ Publish-Subscribe (Publisher ส่งไป Broker กระจายให้ Subscriber ทุกคนที่ Subscribe อยู่)

2. Broker ทำหน้าที่อะไรในระบบ MQTT?
ดูเฉลย

Broker เป็นตัวกลางที่ รับข้อความจาก Publisher จากนั้นกรองและ ส่งต่อให้ Subscriber ที่ Subscribe Topic นั้นอยู่ นอกจากนี้ยังจัดการ authentication และเก็บข้อความสำหรับ Subscriber ที่ออฟไลน์

3. Topic home/+/temp จะรับข้อความจาก Topic ใดได้บ้าง?
ดูเฉลย

home/kitchen/temp ✅, home/bedroom/temp ✅ — แต่ไม่รับ home/floor1/room2/temp ❌ เพราะ + แทนได้แค่ 1 ระดับ

4. QoS ระดับ 2 แตกต่างจาก QoS 0 อย่างไร?
ดูเฉลย

QoS 0 ส่งครั้งเดียวโดยไม่ยืนยัน (อาจหาย) ส่วน QoS 2 ใช้ 4-way handshake เพื่อ รับประกันว่าข้อความถึงผู้รับครั้งเดียวเท่านั้น (Exactly once)

5. Retained Message และ Last Will and Testament (LWT) มีประโยชน์อย่างไร?
ดูเฉลย

Retained Message: Subscriber ใหม่จะได้รับข้อมูลล่าสุดทันทีโดยไม่ต้องรอ Publisher ส่งใหม่
LWT: เมื่ออุปกรณ์หลุดออฟไลน์กะทันหัน Broker จะส่งข้อความที่กำหนดไว้ล่วงหน้าแทน ทำให้ระบบรู้ว่าอุปกรณ์ออฟไลน์

🎓 สรุป

หัวข้อความหมาย
MQTTโปรโตคอลน้ำหนักเบา Publish-Subscribe สำหรับ IoT
Brokerตัวกลางรับและกระจายข้อความ (เช่น Mosquitto)
Publisherอุปกรณ์ที่ส่งข้อมูลไปที่ Broker บน Topic
Subscriberอุปกรณ์ที่รับข้อมูลจาก Broker ตาม Topic ที่ Subscribe
Topicเส้นทางลำดับชั้นคั่นด้วย / เช่น home/room/temp
Wildcard + / #แทน 1 ระดับ / หลายระดับในการ Subscribe
QoS 0/1/2ระดับการรับประกัน: ไม่รับประกัน / อย่างน้อยครั้งเดียว / ครั้งเดียวพอดี
